# Documentation Style Guide (Nefarius House Style)

## Purpose
This repository uses a pragmatic, technically precise documentation style focused on operational clarity, explicit support boundaries, and reproducible setup/build instructions.

## Voice & Tone
- Prefer direct, confident, technical language.
- Avoid marketing fluff and vague claims.
- Use concise prose; prefer bullets for scanability.
- Personality is acceptable in moderation (e.g., in Motivation), but clarity comes first.

## Required README Structure
1. Project title + one-line value proposition
2. Badges/status row (if available)
3. About / Summary
4. Features (bullet list)
5. Limitations / What's missing (if applicable)
6. Supported systems / Environment (explicit OS/arch/version scope)
7. Installation / Quick start
8. Build instructions with prerequisites and exact versions
9. Support policy (issue tracker is not a help desk, where applicable)
10. Licensing + legal/trademark disclaimers (if relevant)
11. Sources/Credits (dependencies, references, related work)

## Writing Rules
- State support boundaries explicitly:
  - supported OS versions
  - supported architectures
  - unsupported scenarios
- Prefer deterministic instructions:
  - exact tool names and versions
  - ordered steps
  - concrete commands
- Distinguish end-user guidance from contributor/build guidance.
- Use tables for compatibility matrices where useful.
- Use "TL;DR" only for complex projects.

## Support & Scope Language
- Be explicit about what is supported and what is not.
- Encourage reading docs/FAQ before filing issues.
- Keep policy statements unambiguous and enforceable.

## Quality Bar
A README is complete only if a new user can answer:
- What does this project do?
- Is my system supported?
- How do I install/use it?
- How do I build it?
- Where do I go for support?
- What legal/licensing constraints apply?
